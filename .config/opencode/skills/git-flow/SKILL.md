## When to use me
- Creating a git commit
- Creating or drafting a pull request
- Creating a new branch

---

## Tool precedence

When performing GitHub operations, use the following order:

1. **GitHub MCP** (preferred) ‚Äî use `github_*` tools (e.g. `github_create_pull_request`, `github_push_files`, `github_create_branch`)
2. **`gh` CLI** ‚Äî fallback if MCP tools are unavailable
3. **Ask the user** ‚Äî if neither MCP nor `gh` is available, ask the user how to proceed before doing anything else

## Commits

Write a short, direct description in sentence case. Describe **what was done**, not which files changed. One line only.

‚úÖ `Created /locale command to set locale of a guild`
‚úÖ `Fixed crash when adding to basket`
‚ùå `Modified RemoteConfigProviderImpl.kt` ‚Äî lists files
‚ùå `fix: apply formatting` ‚Äî no type prefix needed

## Pull Requests

### Title
A short sentence describing what was done (past tense).

‚úÖ `Created /locale command to set locale of a guild`
‚úÖ `Fixed crash when adding to basket`
‚ùå `Modified files for locale` ‚Äî vague, doesn't describe what was done

### Description
Follow the project's `.github/PULL_REQUEST_TEMPLATE.md`. Include:
- **Solves #issue** ‚Äî link the issue(s) this PR closes
- **Changelist Summary** ‚Äî one-liner of what was done
- **Description** ‚Äî more detail about the implementation
- **Steps to reproduce** ‚Äî only include if fixing a bug
- **Media** ‚Äî include if visual changes
- **Extra info** ‚Äî anything else relevant

Trim empty sections.

### AI-generated PR rules
- Open as **Draft**
- Append the following disclaimer at the bottom of the description:

## Branch naming

| Type | Format | Example |
|---|---|---|
| Feature | `feature/{issue}-kebab-desc` | `feature/60-command-to-set-locale` |
| Bug fix | `bugfix/{issue}-kebab-desc` | `bugfix/42-fix-crash` |
| Chore | `chore/{issue}-kebab-desc` | `chore/99-update-deps` |

- Use kebab-case
- Keep description short (3-6 words max)
- Always include the issue number from GitHub Issues

---

## Rebase & amend policy

- Never rebase or amend by default
- If the user requests a rebase, ask for confirmation first and warn about history rewriting
- Only proceed after the user confirms
